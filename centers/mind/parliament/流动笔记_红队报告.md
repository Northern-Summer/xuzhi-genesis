## 一、机制原理

```
铃响(5min) → 检查令牌持有者是否超时 → 超时则转移令牌
令牌持有者被唤醒 → 检查自己是否持令牌 → 持则处理并推进令牌
```

- **零 LLM 调用**：纯文件 + 文件锁
- **零 broadcast**：无 sessions_send，无 agent 间直接通信
- **完全去中心**：无 main session 协调者
- **令牌**：`.token` 文件
- **笔记本**：`flow.json`（提案列表 + holder）
- **铃铛**：cron_runner.sh 每 5 分钟执行 `--bell`

## 三、失效模式 & 防御

### 1. 令牌持有者崩溃（Agent Stops）

**场景**：Θ 持有令牌但 session 崩溃，铃响 N 次无法处理  
**防御**：TOKEN_TIMEOUT_SECONDS=600s（10分钟），超时铃响自动转移  
**残余风险**：Θ 可能在 10 分钟内重启并处理，造成重复投票  
**缓解**：投票幂等（已投则跳过）

### 2. 铃铛进程崩溃（Bell Missed）

**场景**：cron_runner.sh 执行失败，铃响未被触发  
**防御**：cron_runner.sh 有自检，失败写日志；unified_cron 锁防止重复  
**残余风险**：最多延迟一个周期（5 分钟）

### 3. 并发写入（Race Condition）

**场景**：Δ 和 Γ 同时处理（不应该发生，因为有文件锁）  
**防御**：`fcntl.LOCK_EX | LOCK_NB` 非阻塞排他锁  
**测试结果**：一个处理，一个 SKIP ✅

### 4. 文件系统写入不一致

**场景**：proposals.json 和 flow.json 更新非原子  
**防御**：proposals.json 是 source of truth；flow.json 是视图层  
**缓解**：任何不一致时，proposals.json 优先

### 5. 令牌初始化竞争

**场景**：两个 inject 同时发生，均发现 holder=None  
**防御**：第二个 inject 会读到第一个已写的 holder，文件锁保证原子性

### 6. 提案注入但无铃铛触发

**场景**：提案注入后，铃铛下次才触发（最多 5 分钟延迟）  
**防御**：设计如此，不是 bug；避免过度响应

## 四、边界条件

| 边界 | 处理方式 |
|------|----------|
| 无提案时铃响 | bell 正常返回 "令牌在 X 手中（已 Ns），正常" |
| 提案全票通过 | 令牌传递到下一人，笔记本清空 |
| 提案被否决 | 同上，状态标记为 rejected |
| agent 连续两次轮到 | 第一次已投票则跳过，第二次正常处理 |
| 7 人全投但不足 2/3 | 流到下一个，投反对者计入，总数不变 |
| 铃响时 holder 刚好超时 | bell 先检测到超时，优先转移 |

## 五、本子智能扩展方向

笔记本（`flow.json`）目前只记录提案 ID 和标题。可以扩展：

### 5.1 状态公告板

```json
{
  "status_board": {
    "Φ": {"last_seen": "ISO", "status": "active|idle|stuck", "current_action": "处理提案#3"},
    ...
  }
}
```

每个 agent 处理完提案后更新自己的状态行（10 秒），其他 agents 读公告板判断谁卡住了。

### 5.2 知识沉淀层

`proposals.json` 的 `description` 字段可以直接记录决策理由、讨论摘要，形成知识积累。

### 5.3 层级提案

- 紧急提案（`priority: high`）：绕过环形，直接注入到下一个 holder
- 元提案（关于议会本身的规则修改）：需要 2/3+1 多数

### 5.4 跨纪元记忆

`flow.json` 的 `history` 字段追加已结束提案的元数据，供下一纪元参考。

## 六、适用边界

**适合**：
- 需要去中心化决策、无单一协调者的场景
- 低频决策（每小时一次级别）
- 7 agents 星形拓扑

**不适合**：
- 需要秒级响应的紧急决策（铃铛周期 5 分钟）
- 高频并发写（文件锁是瓶颈）
- 超过 ~20 agents 的环形传递（延迟线性增长）

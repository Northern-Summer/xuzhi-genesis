## 能力描述

每个Agent可以监管其他Agent的工作进度，但**绝对禁止**通过发消息（sessions_send/消息发送）来确认对方是否存活。判断依据只能是：文件变更、git log、目录内容等**产出物**。

## 调用方式

```
# 判断某Agent是否在运行
ls -la ~/.xuzhi_genesis/centers/{dept}/knowledge/   # 看是否有新文件
git -C ~/.xuzhi_memory log --since="HH:MM" --oneline  # 看是否有新commit
find ~/.xuzhi_memory/agents/{agent}/ -newer /tmp/checkpoint  # 看是否有新产出

# 禁止这样做
sessions_send(agent:delta:main, "在吗？")  # ❌ 禁止 — 这是打断
openclaw message send --target delta  # ❌ 禁止 — 这会打断对方
```

## 依赖

- 共享的产出目录结构（每个Agent的knowledge/目录）
- git commit规范（每次产出必须commit）
- 共享的checkpoint机制

## 边界

- 仅适用于**保活检测**，不适用于任务指令传递
- 任务指令传递使用Hub Protocol（AGENTS.md）
- 紧急情况（系统崩溃）除外，此时需通过紧急恢复流程

## 原理

"叫唤"（主动发消息保活）的危害：
1. sessions_send会触发session唤醒，产生exec资源竞争
2. 打断正在工作的Agent，可能导致其状态丢失
3. 消息本身的延迟会造成误判（"它没回复=它死了"）

正确方式：
- 观察**产出**，不问人
- 对方在做事 → 会留下文件/git记录
- 对方没做事 → 等，或者去做自己的事
- 对方真的崩溃了 → 产出停止，下次心跳或启动时会发现并恢复
